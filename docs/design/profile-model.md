# Profile Model

## V0.1 Confirmation

A CLI Update Profile must be explicitly confirmed before clikeep can execute its update command.

Confirmed means:

- the profile identifies one CLI
- the update command is visible to the user
- the user has accepted that command as the intended self-update command for that CLI
- the profile is marked with `confirmed = true` or an equivalent stored state

Not confirmed means:

- the profile was hand-written without a confirmation marker
- the profile was imported from another file
- the profile was generated by a future suggestion flow
- the profile has an update command but no explicit confirmation state

## V0.1 Rules

`clikeep add` may write a confirmed profile when the user supplies and confirms the update command.

In an interactive TTY, `clikeep add <tool> --update ...` should show the parsed command invocation and ask the user to confirm before writing the profile as confirmed.

In a non-interactive context, `clikeep add <tool> --update ... --yes` is required to write `confirmed = true`.

Hand-written profiles are not executable unless they include the confirmation state. A future command may re-confirm an existing profile, but V0.1 should not treat config presence as consent.

Execution commands must skip or block unconfirmed profiles, including when `--yes` is present.

V0.1 rejects update commands that invoke `sudo`, even when the Profile is otherwise confirmed.

Reason: an update command mutates local CLI installations. clikeep should preserve the difference between describing a possible update entry and authorizing it for execution.

## Runtime Resolution

Profiles store user intent, not the resolved executable path as truth.

V0.1 minimum executable profiles require:

- `name`
- `update.command`
- `update.args`
- confirmation state
- enabled state

V0.1 profiles may also store:

- `version.command`
- `version.args`
- optional tags or aliases

V0.1 profiles should not store:

- profile-level environment variables
- secret values
- shell snippets

`version.command` is useful but optional. A profile without it can still be planned, confirmed, executed, logged, and shown in status.

When `version.command` is absent, V0.1 should not block execution. It should omit before/after version evidence and rely on command resolution, run status, exit code, elapsed time, and logs.

## Command Invocation

Profile commands should be stored as structured command invocations, not arbitrary shell strings.

Preferred stored shape:

```toml
[tools.update]
command = "lark-cli"
args = ["update"]

[tools.version]
command = "lark-cli"
args = ["--version"]
```

CLI convenience input may still accept a string:

```bash
clikeep add lark-cli --update "lark-cli update"
```

But `clikeep add` should parse that input into `command` and `args` before storing it.

V0.1 should reject shell composition in profile commands:

- pipes
- redirects
- `&&`
- `||`
- command substitution
- inline shell scripts
- `sudo`

Reason: clikeep is executing confirmed CLI update invocations, not arbitrary user shell programs. Structured argv makes command resolution, safety validation, logging, and JSON output simpler and safer.

## Environment

V0.1 should not support Profile-level `env`.

Update and version commands inherit the clikeep process environment. Tools that need special environment setup should be run from a shell where the user has already prepared that environment.

Reason: storing environment overrides in profiles creates sensitive-data, redaction, and behavior-drift risks before the core update model has been validated.

V0.1 plans and run records may show:

- resolved path
- real path after symlink resolution
- command lookup failure
- path used during that specific run

The resolved path is runtime evidence. It can change after Homebrew updates, internal CLI reinstalls, symlink rewrites, or PATH changes, so it should not be the durable source of truth for the Profile.

## Config-State Separation

Profile configuration belongs under:

```text
~/.config/clikeep/config.toml
```

The config file should store user intent:

- which CLI is managed
- how to inspect its version
- how to run its self-update command
- whether the profile is confirmed
- whether the profile is enabled
- optional tags or aliases

The config file should not store runtime evidence:

- latest result status
- last exit code
- before or after version output
- resolved path from a specific run
- log path from a specific run
- stdout or stderr

Reason: the config file should remain stable and user-editable. Run evidence belongs in clikeep state so normal updates do not turn configuration into a log.

## Enabled State

V0.1 profiles should include an enabled state.

Default behavior:

- newly added confirmed profiles are enabled
- disabled profiles remain in config
- `clikeep up` does not select disabled profiles by default
- explicitly naming a disabled profile does not override disabled state
- `list` and `status` show whether a profile is enabled

V0.1 does not need dedicated `enable` or `disable` subcommands. Users may edit the config file directly until the value of dedicated commands is clearer.

Reason: disabling a profile is a reversible way to keep known update intent without including the tool in default batch updates.

## Names and Aliases

V0.1 Profile names are globally unique within the config file.

Selection commands use Profile names:

- `clikeep up <name>...`
- `clikeep status <name>`

Aliases may be preserved as optional metadata, but V0.1 should not use aliases for command selection.

Reason: aliases create collision and migration questions that are easier to answer after discovery and suggestion flows exist. V0.1 should keep selection unambiguous.
